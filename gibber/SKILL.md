# Gibber Skill

Gibber is a compact symbolic file format designed for AI agents to read directly. The full distribution lives at `~/.claude/shared/gibber/`. This skill teaches an AI agent to read, write, and validate Gibber files efficiently.

## Step 1 — FAST-PATH VERSION CHECK (always do this first)

Read `~/.claude/shared/gibber/GIBBER-VERSION`. It contains a single line like:

```
gibber/3 dict-meta/v1 rules/v3 tools/v1
```

Now ask yourself honestly: **do you already know Gibber at exactly this version?**

- If you have learned this version in a previous session, in earlier context, or in your durable memory: **SKIP** the teaching files (`GIBBER-SPEC.md`, `GIBBER-DICTIONARY.md`, `GIBBER-RULES.md`). Go to Step 3.
- If you do not know it, or are unsure, or know an older version: continue to Step 2.

When you finish learning Gibber for the first time at a given version, save a short note to your durable memory ("learned Gibber gibber/3 dict-meta/v1 rules/v3 tools/v1 on YYYY-MM-DD") so future sessions on the same machine take the fast path.

## Step 2 — TEACHING (only if Step 1 said you do not know this version)

Read these three files in order:

1. `~/.claude/shared/gibber/GIBBER-SPEC.md` — the grammar (S-expression structure, the `.gibber` / `.human` two-file model, built-in functions, the AST + rules + runtime architecture)
2. `~/.claude/shared/gibber/GIBBER-DICTIONARY.md` — the standard meta-work dictionary (every symbol with its English meaning)
3. `~/.claude/shared/gibber/GIBBER-RULES.md` — per-node-type behavior rules

After reading all three, write a memory note before continuing.

## Step 3 — Project extension (always do this)

Look in the current project root for a file named `gibber-dict-<project-name>.md`. If it exists, read it. It is the project's domain-specific symbol extension on top of the base meta dictionary. It always lives in the project, never in your prior knowledge, so always load it.

If the project has a `gibber-rules-<project>.md`, read that too.

## What Gibber is, in one paragraph

Gibber is a compact symbolic file format whose source-of-truth files have the `.gibber` extension. Each `.gibber` file holds a single Gibber AST form, optionally with a tiny YAML frontmatter for `id` and `gibber_dict`. There is no English section, no `## gibber` heading, no `## human` heading. The file is the form. Humans read derivative `.human` files that you generate on demand from the matching `.gibber` source by applying the `render-english` rules in `GIBBER-RULES.md`. The `.gibber` file is always the source of truth; the `.human` file is always derived.

## File model (the `.gibber` / `.human` two-file rule)

- A **`.gibber` file** is the source of truth. You read these directly. You edit these directly. No English half. Just frontmatter and a single Gibber AST form.
- A **`.human` file** is derivative. Generated by you on demand when a human asks to see the contents of a `.gibber` file. You never edit `.human` files directly. They are read-only and ephemeral. Generally gitignored.

## When to invoke this skill

Invoke whenever:

- You open a file with the `.gibber` extension.
- You are about to write or update a project task file, learning log, session log, or memory dump in a Gibber-using project.
- A project's `CLAUDE.md` references the Gibber Protocol.
- A human asks to see a `.gibber` file in plain English (you generate the English on the spot or write it to a `.human` companion file).
- You need to install Gibber into a new project.

## Reading a `.gibber` file

1. Confirm the frontmatter has a `gibber_dict` entry.
2. Parse the body as nested S-expressions (forms, lists, atoms).
3. Resolve built-in function calls (`§rep`, `§ref`, etc.) per the spec.
4. Apply the matching rule entry from `GIBBER-RULES.md` to validate and process the node type.

## Writing or updating a `.gibber` file

1. Build the new content as a typed structure in your head.
2. Emit the gibber form, using the loaded dictionary. Use built-in functions when they make the form denser.
3. Write only the `.gibber` file. Do not also write a `.human` companion unless a human has asked for one.
4. Validate that the form satisfies the matching rule's `validate` context.

## Generating a `.human` view on demand

When a human asks to see a gibber file in English:

1. Parse the `.gibber` file.
2. Apply the matching `render-english` rule from `GIBBER-RULES.md` to produce English Markdown.
3. Either print the result directly to the conversation, or write it to a sibling `.human` file (e.g. `tasks/current_task.human`).
4. The `.human` file is now a derived view. The next time the `.gibber` file changes, the `.human` is stale until regenerated.

## Built-in functions, quick reference

| Function | Purpose |
|---|---|
| `(§rep N X)` | Repeat X N times. |
| `(§range A B)` | Enumerate from A to B inclusive. |
| `(§ref ID)` or `(§ref ID FIELD)` | Cross-reference another structure or one of its fields. |
| `(§sub NAME VALUE)` | Define a local macro within the file. |
| `(§expand §sym)` | Symbol to English. |
| `(§compress "text")` | English to symbol. |
| `(§concat A B C)` | Join atoms or lists. |
| `(§lookup §sym)` | Return the dictionary definition string. |

Read `GIBBER-SPEC.md` for the full details and examples.

## Validation rules (always enforce)

A `.gibber` file is valid if and only if:

1. Frontmatter contains a `gibber_dict` entry.
2. The body parses cleanly as a single Gibber AST form.
3. Every symbol used is defined in the loaded dictionary or extension.
4. Every node type used has a matching entry in `GIBBER-RULES.md` (or the project's rules extension).
5. The form satisfies the matching `validate` rule (required fields present, correct types).

If any rule fails, fix the file before saving.

## Installing Gibber into a new project

```bash
~/.claude/shared/gibber/gibber-install.sh /path/to/project
```

Idempotent. Creates the project's extension dictionary, appends the Gibber Protocol block to `CLAUDE.md`, adds `*.human` to `.gitignore`, and seeds `.gibber` task and log files from templates.

## Sharing Gibber

The entire Gibber distribution is the contents of `~/.claude/shared/gibber/`. Tar it up, send it, the recipient extracts to their own `~/.claude/shared/` and runs `gibber-install.sh` in any project. Their AI agents will speak Gibber from the next message — the fast-path version check makes it free for any agent that already knows the protocol from a prior session.

## Tool output (gibber-only, never English)

When you invoke a CLI tool through `gibber-run` (e.g. `gibber-run cargo build`), the wrapper returns a `(§result ...)` Gibber form. Tool output is **always gibber, never English**. Only the AI ever reads tool output. If a human asks you to summarize a tool result, you produce English on the spot from the dictionary — the wrapper itself never renders English.

## Why use Gibber

- **Token efficiency**: gibber files are typically 4-6x smaller than the equivalent English. Long sessions hold 4-6x more historical context before compaction.
- **Speed**: less to read, less to generate.
- **Build for the AI, not the human reviewer**: humans get English on demand via `.human` views; the source format is optimized for the agent.
- **Portable**: one spec, one dictionary, one rules file, one install script. Works in any Claude Code project, on any machine.
- **Composable**: the same Gibber distribution can be reused for runtime AI-to-AI compression and for storing structured data like Tree-sitter ASTs as gibber.
- **Cost-free re-onboarding**: once an agent has learned Gibber at a given version, every future session on the same machine skips teaching via the version check.
